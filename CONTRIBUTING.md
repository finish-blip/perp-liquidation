# 贡献与合并门禁

本服务处理清算执行，任何改动都必须保留风控授权边界、幂等、顺序、`reduce_only`、fencing token、成交后等待结算和 Outbox 可靠投递等不变量。

## 开发流程

1. 从最新 `main` 创建短生命周期分支。
2. 为行为变化增加或更新测试。
3. 执行 [`docs/testing.md`](docs/testing.md) 中的本地门禁。
4. 通过 Pull Request 合并，不直接推送到受保护的 `main`。
5. 至少由一名熟悉风控、撮合、仓位或结算契约的维护者审核。

## Pull Request 要求

- 说明行为变化、失败模式和回滚方式。
- 标出数据库迁移、接口字段、事件 Schema 和环境变量变化。
- 不提交凭据、Token、真实账户数据、生产日志或本地依赖目录。
- 不以降低测试强度、扩大重试副作用或绕过前置校验的方式修复失败。
- 接口变更必须保持向后兼容，或提供明确的版本升级顺序。
- 安全扫描例外必须记录受影响版本、攻击面、补偿控制、负责人和截止日期。

## 必须保持的不变量

- 重复风险决策不得重复下单。
- 旧决策序号不得产生外部副作用。
- 执行数量不得超过风控授权或当前可减仓位。
- 行情过期、价格越界或仓位版本不匹配时不得下单。
- 下单结果不确定时必须先按 `client_order_id` 反查。
- `FILLED` 不等于完成，必须等待仓位/账务结算确认。
- 结果先写 Outbox，再异步投递给风控。
